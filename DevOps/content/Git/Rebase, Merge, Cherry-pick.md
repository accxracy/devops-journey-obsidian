# 🔀 Слияние кода: Merge vs Rebase

Ты создал ветку `feature` от `main`, написал код. В это время кто-то другой (или ты сам) запушил новые фиксы в `main`. Твоя ветка отстала. Тебе нужно обновить `feature` свежими данными из `main`, чтобы не было конфликтов. У тебя два пути:

## 1️⃣ `git merge` (Безопасно, но грязно)
Принцип "слепить дороги вместе".

* **Как работает:** Git берет текущее состояние `main`, берет твою `feature` и создает **абсолютно новый коммит** (Merge Commit), у которого два родителя. 
* **Команда:** Находясь в ветке `feature`, пишешь `git merge main`.
* **Плюсы:** Безопасно. История коммитов остается на 100% достоверной (видно, кто, когда и откуда отпочковался).
* **Минусы:** Граф истории превращается в нечитаемую "схему метро" (спагетти из линий), если над проектом работают 10 человек.

## 2️⃣ `git rebase` (Красиво, но опасно)
Принцип "переписать историю".

* **Как работает:** Git отрывает твои новые коммиты от `main` (от того места, где ты ветвился), скачивает свежий `main`, и **приклеивает твои коммиты на самый верх** свежего `main`. 
* Физически Git создает твои коммиты заново (у них меняются хэши). 
* **Команда:** Находясь в ветке `feature`, пишешь `git rebase main`.
* **Плюсы:** История выглядит как идеальная прямая линия (Линейная история). Идеально для энтерпрайза.
* **Минусы (Золотое правило Rebase):** ⚠️ **НИКОГДА не делай rebase публичных веток (например, `main`)!** Ты перепишешь историю (изменишь хэши). Если другие разработчики уже скачали старый `main`, при попытке запушить у них всё взорвется от конфликтов. Rebase делают **только** для своих локальных, еще не запушенных веток-фич.

## 🍒 `git cherry-pick` (Точечный перенос)
Представь, что в ветке `dev` есть 5 коммитов. Один из них — это гениальный багфикс. Тебе нужен только этот багфикс в ветку `main`, но остальные 4 коммита (недоделанные фичи) сливать в `main` пока нельзя. `merge` и `rebase` тут не помогут.

* **Как работает:** Берет копию *одного конкретного коммита* и применяет его изменения к твоей текущей ветке как новый коммит.
* **Команда:** 
 1. Находишь хэш нужного коммита (например, `a1b2c3d`).
 2. Переключаешься в ветку, куда хочешь его вставить: `git checkout main`.
 3. Пишешь: `git cherry-pick a1b2c3d`.

## ⚔️ Конфликты (Merge Conflicts)
Конфликт возникает, когда два человека изменили **одну и ту же строчку** в одном файле. Git останавливает слияние и говорит: "Я тупой робот, я не знаю, чей код правильный. Решай сам".

1. Открываешь проблемный файл в VS Code. Ты увидишь маркеры Гита:

  <<<<<<< HEAD (Твоя текущая ветка)
  print("Hello, DevOps!")
  =======
  print("Hello, World!")
  >>>>>>> feature-branch (Ветка, которую ты вливаешь)

2. Твоя задача: удалить маркеры `<<<<`, `====`, `>>>>` и оставить ту версию кода, которая правильная (или написать третью, объединив их логику).
3. Сохранить файл.
4. Добавить его в индекс: `git add <file>`.
5. Завершить процесс: `git commit` (или `git rebase --continue`, если был в процессе ребейза).
