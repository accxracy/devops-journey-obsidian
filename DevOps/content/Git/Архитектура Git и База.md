# 🐙 Архитектура Git: Как это работает под капотом

Главная ошибка новичков — они думают, что Git сохраняет только *изменения* (дельты) в файлах (типа "в строке 5 добавлено слово"). 
**Это не так.** Git работает как фотоаппарат. Каждый коммит — это полный **слепок (snapshot)** всей твоей папки проекта в конкретную миллисекунду. Если файл не менялся, Git просто делает ссылку на его предыдущую версию, чтобы не тратить место.

## 🏗 Три зоны Git (The Three States)
Это фундамент. Твой локальный проект всегда разделен на 3 виртуальные зоны. Непонимание этих зон — причина 90% проблем с Git.

1. **Working Directory (Рабочая директория):** Это то, что ты видишь в своей IDE (VS Code). Твои реальные файлы. Ты пишешь код здесь.
2. **Staging Area / Index (Зона подготовки):** Виртуальная "сцена". Представь, что ты собираешь коробку для переезда. Ты кладешь туда файлы по одному. Коробка еще не запечатана, ты можешь что-то вытащить.
3. **Repository (Локальный репозиторий / `.git`):** Ты заклеил коробку скотчем, написал на ней дату и положил на склад. Это и есть **Коммит**. Он неизменен (почти).

### 🔄 Жизненный цикл файла (Базовые команды)
* `git status` : Посмотреть, в каких зонах сейчас находятся файлы.
* `git add <file>` : Переместить файл из Working Directory в Staging Area (положить в коробку).
* `git commit -m "Fix bug"` : Запечатать коробку. Перенести всё из Staging в Repository. Создается уникальный хэш (например, `a1b2c3d`).
* `git push` : Отправить запечатанные коробки с твоего локального склада на удаленный сервер (GitHub).

## 🌳 Ветки (Branches) и Указатели (HEAD)

Вторая ошибка новичков: они думают, что Ветка — это физическая папка с копией кода.
В Git **Ветка — это просто текстовый файл, в котором записан хэш одного коммита**. Это плавающий стикер (ярлык).

Когда ты делаешь новый коммит, стикер ветки (например, `main`) просто переклеивается на этот новый коммит.

### Что такое `HEAD`?
У Git есть спец-указатель `HEAD` (пишется капсом). Это "глаз" Гита. Он показывает, **где ты сейчас находишься**.
* Обычно `HEAD` приклеен к имени текущей ветки. 
* Если ты переключаешься на другую ветку (`git checkout dev`), `HEAD` перепрыгивает на ветку `dev`, и файлы в твоей рабочей папке мгновенно меняются на те, что лежат в коммите `dev`.

### Отсоединение головы (Detached HEAD)
Страшная ошибка, которую часто ловят джуны. Возникает, когда ты делаешь `git checkout <хэш_старого_коммита>`, а не `git checkout <имя_ветки>`.
* `HEAD` отрывается от ветки и прилипает напрямую к старому коммиту. 
* Если ты в этот момент напишешь код и сделаешь `commit`, этот коммит не будет принадлежать ни одной ветке. Как только ты переключишься обратно на `main`, этот новый код "потеряется" в пустоте (его сожрет Garbage Collector Гита).
* *Решение:* Если попал в Detached HEAD и написал код — срочно создай ветку прямо здесь: `git checkout -b rescue-branch`.

## 🛠 Памятка: Машина времени
* `git log` : Посмотреть историю коммитов (ленту слепков).
* `git checkout <branch>` : Перепрыгнуть на другую ветку.
* `git checkout -b <new_branch>` : Создать новую ветку и сразу перепрыгнуть на нее.
