# 🛡 Nginx: Базовая архитектура и Reverse Proxy

**Nginx** — это высокопроизводительный веб-сервер. В современном DevOps его главная роль — работать как **Reverse Proxy (Обратный прокси)**.

## 🔄 Что такое Reverse Proxy?
Обычно клиент (браузер) напрямую подключается к твоему приложению (например, Node.js на порту `8080`). 
В схеме с Reverse Proxy между интернетом и приложением встает Nginx.

1. Nginx "слушает" публичные порты `80` (HTTP) и `443` (HTTPS).
2. Принимает все входящие запросы.
3. Анализирует доменное имя (URL) из запроса.
4. Перенаправляет запрос на нужный внутренний сервер или Docker-контейнер (например, на `127.0.0.1:3000`).

**Зачем это нужно:**
* **Балансировка нагрузки:** Nginx может распределять трафик между 10 одинаковыми контейнерами.
* **Безопасность:** Внутренние приложения скрыты от прямого интернета. Nginx отбивает базовые DDoS-атаки и фильтрует кривые запросы.
* **Единая точка входа:** Можно хостить 5 разных сайтов на одном сервере, и Nginx сам разберется, кому какой трафик отдать.

## 🏗 Архитектура конфигов (Где что лежит)
В Linux (Debian/Ubuntu) все настройки Nginx лежат в директории `/etc/nginx/`.

* **/etc/nginx/nginx.conf** — Глобальный файл настроек. Задает общие параметры (лимиты, настройки логирования, количество рабочих процессов). Его меняют редко.
* **/etc/nginx/sites-available/** — Папка, где лежат конфигурационные файлы для каждого отдельного сайта/приложения (например, `bot.conf`, `grafana.conf`).
* **/etc/nginx/sites-enabled/** — Папка с симлинками (ярлыками). **Nginx читает только конфиги из этой папки**.
 * *Паттерн DevOps:* Ты создаешь файл в `sites-available`, а затем делаешь на него симлинк в `sites-enabled`. Чтобы временно "выключить" сайт, ты просто удаляешь ярлык из `sites-enabled` и перезапускаешь Nginx. Файл настроек при этом сохраняется в `sites-available`.

## 📄 Базовый синтаксис (Пример Reverse Proxy)

Как выглядит классический конфиг для проксирования трафика на внутренний Docker-контейнер (порт `3000`):

 server {
  # 1. Порт, который слушает Nginx
  listen 80;
  
  # 2. Имя домена, на которое реагирует этот блок
  server_name my-app.com www.my-app.com;

  # 3. Маршрутизация (куда отправлять трафик)
  location / {
  # Перенаправляем трафик на наше приложение
  proxy_pass http://127.0.0.1:3000;
  
  # Обязательные заголовки, чтобы приложение видело реальный IP юзера
  proxy_set_header Host $host;
  proxy_set_header X-Real-IP $remote_addr;
  }
 }

## 🛠 Полезные команды (CLI)

* `nginx -t` : **Главная команда**. Проверяет синтаксис всех конфигурационных файлов. Всегда делай это перед перезапуском! Если там опечатка, команда покажет в какой строке ошибка, и не даст Nginx упасть.
* `systemctl reload nginx` : Мягкая перезагрузка. Применяет новые конфиги, но не разрывает текущие соединения пользователей (Graceful reload).
* `systemctl restart nginx` : Жесткий перезапуск процесса Nginx.
* `systemctl status nginx` : Посмотреть, работает ли служба и нет ли ошибок в ядре.
