# ☸️ Kubernetes: Архитектура кластера

Kubernetes (K8s) работает как государство. Есть правительство (Control Plane), которое принимает решения, и есть рабочие (Worker Nodes), которые выполняют физическую работу по запуску контейнеров.

## 🧠 Control Plane (Мастер-нода)
Это мозг кластера. Здесь не крутятся приложения пользователей.
* **kube-apiserver:** Единая точка входа. Все команды (от админа через `kubectl` или от внутренних компонентов) идут только через API-сервер. 
* **etcd:** База данных кластера (Key-Value хранилище). Единственное место, где K8s хранит свое состояние (state). Если умрет `etcd` — кластер забудет, кто он такой.
* **kube-scheduler:** Распределитель. Видит, что нужно запустить новый контейнер, ищет свободный Worker-сервер (по RAM и CPU) и назначает задачу туда.
* **kube-controller-manager:** Автоматика самолечения. Бесконечный цикл, который следит, чтобы текущее состояние кластера совпадало с желаемым (декларативным). Умер сервер? Контроллер заметит это и попросит создать новые контейнеры.

## 👷‍♂️ Worker Nodes (Рабочие узлы)
Это обычные сервера (VPS/железо), на которых крутятся сами приложения.
* **kubelet:** "Прораб" на каждом сервере. Слушает API-сервер. Если ему говорят "запусти контейнер" — он идет к Docker (или containerd) и запускает его.
* **kube-proxy:** Сетевой маршрутизатор. Настраивает правила `iptables` на сервере так, чтобы сетевой трафик доходил до нужных контейнеров.
* **Container Runtime:** Непосредственно движок, который крутит контейнеры (Docker, containerd, CRI-O).
