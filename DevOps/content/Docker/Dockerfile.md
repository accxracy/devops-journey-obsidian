# 🥞 Dockerfile: Инструкции и Архитектура слоев

**Dockerfile** — это сценарий (манифест), по которому собирается Docker-образ. 

Главное правило Докера: **Каждая инструкция в Dockerfile создает новый физический слой (Layer).**
Слой неизменяем (Read-Only). Если в слое №1 ты скачал файл 1 ГБ, а в слое №2 удалил его — итоговый образ всё равно будет весить 1 ГБ, потому что файл навсегда остался "замороженным" в первом слое.

## 🛠 Основные инструкции

* `FROM <образ>:<тег>` : **База.** Всегда первая строчка. Указывает, какой готовый образ взять за основу (например, `FROM python:3.11-slim`).
* `WORKDIR /app` : **Папка.** Создает рабочую директорию внутри контейнера и переходит в нее (как команда `cd /app`).
* `COPY <откуда> <куда>` : **Копирование.** Переносит файлы с твоего хост-сервера внутрь образа.
* `ENV <ключ>=<значение>` : **Переменные.** Задает переменные окружения, которые будут доступны приложению.

## ⚔️ Главный вопрос: `RUN` vs `CMD`

Эти две команды делают одно и то же — выполняют код в терминале. Но делают это на **фундаментально разных этапах** жизни Докера.

### 1️⃣ `RUN` (Выполняется во время СБОРКИ)
* **Когда работает:** В тот момент, когда ты пишешь `docker build`.
* **Зачем нужен:** Чтобы **подготовить** образ. Установить библиотеки, скомпилировать код, скачать утилиты.
* **Результат:** То, что сделал `RUN`, "запекается" в новый слой образа навсегда.
* *Пример:* `RUN pip install -r requirements.txt` или `RUN apt-get update && apt-get install -y curl`.

### 2️⃣ `CMD` (Выполняется во время ЗАПУСКА)
* **Когда работает:** В тот момент, когда ты пишешь `docker run`.
* **Зачем нужен:** Это **главная цель** контейнера. То, ради чего он вообще был создан. `CMD` не создает слоев, он просто говорит: "Когда меня запустят, включи эту программу".
* **Особенности:**
 * В Dockerfile может быть только **ОДИН** `CMD`. Если напишешь несколько — сработает только последний.
 * Если программа, запущенная через `CMD`, завершит свою работу — контейнер немедленно умрет.
* *Пример:* `CMD ["python", "bot.py"]` или `CMD ["nginx", "-g", "daemon off;"]`.

*(Есть еще `ENTRYPOINT`. Он работает как `CMD`, но его нельзя случайно перезаписать при запуске контейнера с параметрами. Девопсы часто объединяют их: `ENTRYPOINT` запускает скрипт, а `CMD` передает в него аргументы).*

## ⚡️ Магия кэширования (Как не ждать сборку полчаса)

Docker собирает образы лениво. Если инструкция не изменилась, он берет готовый слой из кэша. 
⚠️ **Золотое правило:** Как только ломается кэш одного слоя, все слои, идущие *ниже*, пересобираются с нуля!

**❌ Как делают джуны (Плохо):**

 FROM python:3.11
 WORKDIR /app
 COPY . .   
 RUN pip install -r req.txt 
 CMD ["python", "main.py"]

*Проблема:* Ты изменил опечатку в `main.py`. Инструкция `COPY . .` видит изменения и ломает кэш. Инструкция `RUN pip install` вынуждена запускаться заново и 5 минут качать пакеты, хотя они не менялись.

**✅ Как делают инженеры (Хорошо):**

 FROM python:3.11
 WORKDIR /app
 COPY requirements.txt . 
 RUN pip install -r req.txt 
 COPY . .   
 CMD ["python", "main.py"]

*Результат:* Ты меняешь код. Первые инструкции берутся из кэша за 0.01 сек. Выполняется только быстрая инструкция копирования самого кода. Образ собирается моментально.


## `RUN` vs `CMD` vs `ENTRYPOINT`

* `RUN` — Выполняется **при сборке** образа (`docker build`). "Запекает" результат в новый неизменяемый слой (качает пакеты, компилирует код).
* `CMD` — Выполняется **при запуске** контейнера (`docker run`). Задает команду/аргументы по умолчанию. Легко перезаписывается юзером (например, `docker run my_image bash` заменит дефолтный `CMD`). В файле может быть только один `CMD`.
* `ENTRYPOINT` — Тоже выполняется **при запуске**, но жестко фиксирует исполняемый файл. Юзер не может его перезаписать просто так. То, что юзер пишет в конце `docker run`, **передается как аргумент** внутрь `ENTRYPOINT`.

**💎 Best Practice:** Использовать `ENTRYPOINT` и `CMD` вместе. 
`ENTRYPOINT` жестко задает саму программу (например, `["nginx"]`), а `CMD` задает дефолтные флаги (`["-g", "daemon off;"]`), которые пользователь при желании может легко заменить на свои (например, на флаг `-t` для теста конфига).
