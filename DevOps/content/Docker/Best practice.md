# 🏆 Docker Best Practices (Оптимизация и ИБ)

Написать Докерфайл, чтобы «просто работало» может любой. Девопс (и DevSecOps) пишет так, чтобы образ был легким, быстрым и безопасным.

## 🛡 1. Безопасность (Security Hardening)
* **Никакого Root:** По умолчанию процессы в контейнере работают от пользователя `root`. Если хакер найдет уязвимость в твоем боте и пробьет изоляцию (Container Breakout), он получит права рута на всем сервере.
 * *Как надо:* В конце Докерфайла создавать пользователя: 
  `RUN useradd -m appuser`
  `USER appuser`


## 🚀 2. Оптимизация сборки
* **Файл `.dockerignore`:** Работает как `.gitignore`. Туда нужно вписать `venv/`, `.git/`, `__pycache__/`. Иначе команда `COPY . .` затащит весь этот мусор в образ, раздув его на сотни мегабайт и замедлив сборку.
* **Слои:** Группируй команды `RUN` через `&&`, удаляй кэш пакетных менеджеров в том же слое (например, `rm -rf /var/lib/apt/lists/*`).
* **Multi-stage сборка:** Компилируй код в "толстом" образе, а в продакшен отдавай готовый бинарник в пустом Alpine-образе (снижает Attack Surface — поверхность атаки).

## ⚙️ 3. Архитектура приложения
* **Один контейнер = Один процесс.** Не пытайся засунуть в один Докерфайл Nginx, Базу и Python-бота через Systemd или Supervisor. Контейнер должен делать что-то одно. Если процесс умирает, Докер должен это понять и перезапустить именно этот узел, а не всю связку.
* **Логирование:** Приложение внутри Докера никогда не должно писать логи в файл (например, в `bot.log`). Оно должно писать их строго в `stdout/stderr` (обычный `print()`). Докер сам перехватит этот поток, запишет на диск хоста и обеспечит ротацию логов (чтобы диск не забился).

## 🪆 4. Multi-stage сборка (Снижение поверхности атаки)
В обычной сборке в итоговый образ попадает всё: исходный код, компиляторы, приватные SSH-ключи для скачивания библиотек, пакетные менеджеры (`pip`, `npm`, `apt`) и утилиты вроде `curl`. 
Для ИБ это катастрофа: если хакер найдет уязвимость и пробьется в контейнер, у него будет полный набор инструментов (Attack Surface), чтобы скачать из интернета троян или майнер.

**Multi-stage** решает эту проблему физическим разделением сред:
1. **Стадия Builder:** Толстый образ. Содержит компиляторы и исходники. Собирает код.
2. **Стадия Production:** Идеально пустой образ (например, `alpine` или вообще `scratch`). Он просто забирает скомпилированный бинарный файл из первой стадии.

**Пример идеального безопасного образа (Distroless):**

 # Стадия 1: Сборка
 FROM golang:1.20 AS builder
 WORKDIR /app
 COPY . .
 RUN go build -o my_app main.go

 # Стадия 2: Продакшен (Scratch - пустой образ ядра)
 FROM scratch
 COPY --from=builder /app/my_app /my_app
 ENTRYPOINT ["/my_app"]

* **Итог для ИБ:** Итоговый образ весит 10 МБ вместо 800 МБ. Внутри образа `scratch` **вообще нет консоли** (`/bin/bash` или `sh`). Даже если хакер выполнит RCE (Remote Code Execution), он физически не сможет запустить ни одну команду ОС, потому что самой ОС внутри образа нет.
