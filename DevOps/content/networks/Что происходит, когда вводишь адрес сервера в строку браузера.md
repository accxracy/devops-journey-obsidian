# 🌍 Что происходит при вводе `google.com` в браузере?

Это классический вопрос на собеседовании, который проверяет понимание всего стека технологий (от локального кэша до серверной архитектуры). 

Весь процесс делится на 5 жестких этапов:

## 1️⃣ Поиск IP-адреса (DNS Resolution)
Браузер не умеет подключаться к тексту `google.com`. Ему нужен IP-адрес (например, `142.250.190.46`). Чтобы его найти, запускается процесс резолвинга (поиска):

1. **Кэш браузера:** Сначала браузер проверяет свою внутреннюю память.
2. **Кэш ОС (Файл `hosts`):** Если в браузере пусто, ОС Linux смотрит файл `/etc/hosts` (там можно жестко захардкодить IP для доменов).
3. **Локальный роутер:** Запрос уходит на твой домашний Wi-Fi роутер.
4. **Провайдер (ISP) и DNS-сервера:** Если никто не знает IP, запрос уходит к DNS-серверу твоего провайдера (или на сервер `8.8.8.8`). Тот рекурсивно опрашивает корневые сервера интернета, пока не найдет нужный IP, и возвращает его твоему браузеру.

## 2️⃣ Установка соединения (TCP Handshake)
IP-адрес найден. Теперь браузер должен проложить надежный канал связи до сервера (по протоколу TCP).
Происходит "Тройное рукопожатие" (3-way Handshake):
1. **Браузер:** «Привет, сервер, я хочу передать данные, ты готов?» (`SYN`)
2. **Сервер:** «Привет, да, я готов, а ты готов?» (`SYN-ACK`)
3. **Браузер:** «Да, погнали!» (`ACK`)

## 3️⃣ Безопасность (TLS/SSL Handshake)
Если сайт использует `HTTPS` (а сейчас его используют все), поверх TCP происходит еще одно рукопожатие.
* Браузер и сервер обмениваются криптографическими ключами. 
* Сервер показывает свой SSL-сертификат (доказывает, что он реально Google).
* С этого момента весь трафик между ними шифруется. Никакой хакер на роутере не сможет прочитать переписку.

## 4️⃣ Отправка HTTP-запроса и работа сервера (DevOps Зона)
Браузер отправляет зашифрованный текстовый запрос (например, `GET / HTTP/1.1`).
Пакет прилетает на сервер Google. Что происходит внутри (классическая архитектура):

1. **Файрвол (UFW/iptables):** Проверяет, не в черном ли списке IP юзера, и разрешает доступ к 443 порту (HTTPS).
2. **Балансировщик нагрузки (Nginx / HAProxy):** Принимает запрос. Видит, что у Google 1000 серверов, выбирает наименее загруженный и перенаправляет запрос туда.
3. **Бэкенд (Python / Go / Java):** Приложение читает запрос, идет в Базу Данных, достает нужную информацию, генерирует HTML-страницу.

## 5️⃣ Ответ и Рендеринг
Сервер отправляет готовую HTML-страницу обратно браузеру (`HTTP/1.1 200 OK`).
Браузер читает код (HTML, CSS, JS), скачивает картинки (каждая картинка — это отдельный мини-запрос по той же схеме) и отрисовывает страницу на экране. Соединение закрывается.
