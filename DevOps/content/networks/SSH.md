# 🔐 SSH (Secure Shell) и Асимметричная криптография

**SSH (порт 22 / TCP)** — это защищенный протокол для удаленного управления сервером. В DevOps доступ по логину и паролю считается серьезной уязвимостью (его можно сбрутфорсить). Вместо этого используют SSH-ключи.

## 🧠 Как это работает под капотом (Асимметричная криптография)
Когда ты пишешь `ssh-keygen`, математический алгоритм генерирует два связанных ключа:
1. **Приватный ключ (`id_rsa`):** Твой личный "сейф". Никогда и никому не передается, хранится только на твоем компе. Имеет права доступа `600` (читать можешь только ты).
2. **Публичный ключ (`id_rsa.pub`):** Твой "замок". Ты можешь раздавать его кому угодно и копировать на любые сервера.

**Процесс авторизации:**
* Ты копируешь свой *Публичный ключ* на удаленный сервер в файл `~/.ssh/authorized_keys`.
* Когда ты пытаешься зайти, сервер берет случайную строку текста, шифрует ее твоим *Публичным ключом* и присылает тебе.
* Расшифровать это сообщение может **только** твой *Приватный ключ*.
* Твой комп расшифровывает сообщение, отправляет обратно серверу. Сервер видит: "Ага, текст совпал, значит у этого парня есть приватный ключ". И пускает тебя без пароля.

## 🛠 Практика: Базовая настройка SSH в контейнере / на VPS
*(На основе гайда: базовая защита сервера)*

### Шаг 1. Генерация ключей на твоем ПК
Если ключей еще нет, генерируем их командой:
`ssh-keygen -t rsa -b 4096`

Смотрим и копируем наш публичный ключ:
`cat ~/.ssh/id_rsa.pub`

### Шаг 2. Закидываем ключ на сервер (под обычным юзером)
Заходим на сервер, создаем нужные папки и вставляем скопированный ключ:
`mkdir -p ~/.ssh`
`nano ~/.ssh/authorized_keys` *(<--- Вставляем сюда публичный ключ)*

**⚠️ ЖЕСТКО ЗАДАЕМ ПРАВА ДОСТУПА**
Если права будут слишком открытыми, SSH-сервер откажется использовать ключ!
`chmod 700 ~/.ssh`
`chmod 600 ~/.ssh/authorized_keys`

### Шаг 3. Хардэнинг (Защита сервера)
Мы закинули ключ, теперь нужно запретить вход по паролю, чтобы отсечь хакеров-брутфорсеров. Открываем главный конфиг SSH-сервера (нужны права root/sudo):
`sudo nano /etc/ssh/sshd_config`

Меняем следующие параметры:
* `PermitRootLogin no` (Запрещаем заходить сразу под root, только через обычного юзера + sudo)
* `PubkeyAuthentication yes` (Разрешаем вход по нашим ключам)
* `PasswordAuthentication no` (Полностью отключаем вход по паролю!)

### Шаг 4. Применяем настройки
Перезапускаем службу SSH, чтобы она прочитала новый конфиг:
`sudo systemctl restart ssh` (на VPS) 
или `service ssh restart` (внутри Docker-контейнера).

✅ **Готово.** Теперь твой сервер невозможно взломать перебором паролей. Доступ есть только с той машины, на которой лежит приватный ключ.
